# cy-412 智慧社区物业管理系统

请根据以下要求生成项目代码和 README。

## 项目主要需求

### 1. 项目概述

**智慧社区物业管理系统（SmartEstate）** 是一套面向住宅小区物业与业主的数字化管理平台。业主可在线报修、缴纳物业费、查看公告、预约访客；物业人员可处理工单、管理住户、发布公告、统计收费。项目类型：**全栈 Web 应用**。

### 2. 核心实体（≥ 3 个，必须贯穿全栈）

- **User（用户）**
  - 字段：id, phone, password_hash, nickname, avatar, role（业主/物业/管理员）, building, unit, room, created_at
  - 关键关系：一个用户可提交多个 Repair，参与多个 Announcement 的阅读记录
  - 业务动作：注册/登录、修改资料、绑定房产
  - **此实体必须贯穿全栈**

- **Repair（报修工单）**
  - 字段：id, user_id, title, description, type（水电/家具/公共设施/其他）, images, status, handler_id, created_at, updated_at
  - 关键关系：隶属于 User，由物业人员（User.handler）处理
  - 业务动作：提交报修、分配处理人、更新进度、业主评价
  - **此实体必须贯穿全栈**

- **Payment（物业缴费）**
  - 字段：id, user_id, fee_type（物业费/停车费/水电费）, amount, month, status, paid_at, created_at
  - 关键关系：隶属于 User
  - 业务动作：生成账单、在线缴费、查看历史账单
  - **此实体必须贯穿全栈**

- **Announcement（社区公告）**
  - 字段：id, title, content, category（通知/活动/紧急）, publisher_id, publish_at, top, read_count
  - 关键关系：由物业人员发布，所有用户可见
  - 业务动作：发布公告、置顶、查看详情、记录阅读
  - **此实体必须贯穿全栈**

### 3. 核心页面（≥ 4 个，至少 2 个页面共用同一业务组件）

- **/dashboard（物业工作台）**
  - 消费实体：Repair、Payment、Announcement
  - 关键交互：待处理工单数、本月收费统计、最新公告
  - 共享组件：`<StatCard>`、`<RepairStatusBadge>`

- **/repairs（报修管理）**
  - 消费实体：Repair、User
  - 关键交互：工单列表、状态筛选、分配处理人、进度更新
  - 共享组件：`<RepairCard>`、`<RepairStatusBadge>`（与 /dashboard 共用）

- **/payments（费用缴纳）**
  - 消费实体：Payment、User
  - 关键交互：账单列表、在线支付（支付宝沙箱模拟）、缴费记录
  - 共享组件：`<PaymentItem>`、`<EmptyState>`

- **/announcements（社区公告）**
  - 消费实体：Announcement、User
  - 关键交互：公告列表、置顶标识、详情查看、物业发布
  - 共享组件：`<AnnouncementCard>`、`<CategoryTag>`

- **/profile（个人中心）**
  - 消费实体：User
  - 关键交互：修改昵称、头像、绑定房产
  - 共享组件：`<AvatarUploader>`

### 4. 横切关注点（≥ 2 个）

- **RBAC 权限控制**
  - 触达文件层：
    - 数据库：`roles` 表、`permissions` 表、`role_permissions` 关联表
    - 后端：`middlewares/auth.js`、`middlewares/rbac.js`、`services/permissionService.js`
    - 路由：每个路由标注所需权限
    - 前端：`router/guards.ts`、`components/common/PermissionButton.tsx`、`stores/authStore.ts`
    - 类型：`frontend/src/types/permission.ts`

- **操作日志**
  - 触达文件层：
    - 数据库：`operation_logs` 表
    - 后端：`middlewares/operationLog.js`、`services/operationLogService.js`
    - 路由：所有写操作路由附加日志中间件
    - 前端：管理员可查看日志页面

- **接口限流**
  - 触达文件层：
    - 后端：`middlewares/rateLimiter.js`，基于内存或 Redis
    - 配置：`config/rateLimit.ts`
    - 路由：登录、支付等敏感接口启用限流

### 5. 共享枚举/常量（≥ 2 组）

- **RepairStatus（工单状态）**
  - 值：PENDING = 'pending', ASSIGNED = 'assigned', PROCESSING = 'processing', DONE = 'done', CLOSED = 'closed'
  - 前端定义位置：`frontend/src/constants/repair.ts`
  - 后端定义位置：`backend/src/constants/repair.js`
  - 出现位置：Repair 模型、RepairService、RepairCard、RepairStatusBadge、筛选器、日志模板
  - **README 必须列出该枚举在前后端的所有出现位置**

- **UserRole（用户角色）**
  - 值：RESIDENT = 'resident', STAFF = 'staff', ADMIN = 'admin'
  - 前端定义位置：`frontend/src/constants/user.ts`
  - 后端定义位置：`backend/src/constants/user.js`
  - 出现位置：User 模型、RBAC 中间件、路由权限、前端按钮显隐、路由守卫
  - **README 必须列出该枚举在前后端的所有出现位置**

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

1. **日志模块必须单独管理但全栈引用**
   - 后端必须有一个 `utils/LogUtil.java`，所有 controller/service/interceptor 都引用
   - 日志模板集中在 `constants/LogTemplates.java`，包含至少 25 条模板
   - 任何业务字段变更（如 Repair 新增字段）都必须同步修改日志模板

2. **异常信息必须分散且层层透传**
   - 错误码集中在 `common/ErrorCode.java`，但 service 和 controller 都要手动包装
   - 错误 message 必须包含实体名、字段名、当前用户角色，如 `"Repair[id=123] assign failed: staff not found"`

3. **常量/工具类必须多处耦合**
   - `utils/CommonFormatter.java` 同时包含日期格式化、金额格式化、状态文本、角色文本
   - `constants/Messages.java` 同时包含前端提示文案、后端返回文案、日志文案
   - 修改一个文案可能同时影响前端显示、后端返回和日志内容

4. **权限字符串必须分散定义**
   - 权限码在 `constants/Permissions.java`、前端路由 meta、按钮权限码、数据库初始化脚本中同时存在
   - 新增一个权限需要修改至少 6 处

5. **枚举必须前后端重复定义且被多处引用**
   - RepairStatus 和 UserRole 必须在前端 constants、后端 constants、类型定义、日志模板、错误码、数据库枚举/约束中同时存在
   - 新增一个枚举值时，必须修改至少 10 处文件

6. **牵一发动全身的验证标准**
   - 后续如果要求"给 Repair 新增一个 'cancelled' 状态"，必须触达：数据库脚本、Repair 实体、前后端 constants、RepairService 状态机、RepairCard、RepairStatusBadge、日志模板、错误码、formatters、统计卡片等 ≥ 10 个文件

### 12. 部署要求

这是前后端分离项目。

前端代码放在 `frontend/`：
- 页面、组件、路由、状态管理、API 请求、类型定义、样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
- 数据库脚本放在 `database/`。

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=smartestate`
- `frontend/Dockerfile`
- `backend/Dockerfile`
- `frontend/nginx.conf`
- `.env.example`

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: smartestate`
- 所有服务使用 `container_name` 命名容器，容器名带 `${COMPOSE_PROJECT_NAME:-smartestate}` 前缀
- 数据库数据使用命名卷持久化
- 前端通过 Nginx 反向代理后端 API（前端代码里调用 `/api`，由 Nginx 转发到 `backend:<port>`）
- 服务间通过 Docker 内部网络通信
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- 后端服务也配置 `healthcheck`
- 暴露端口：前端 `18412:80`，后端 `19412:8080`，数据库 `3306:3306`
- 所有可变配置通过环境变量注入
- 支持通过 `docker compose up -d` 一条命令启动

`.env` / `.env.example` 必须包含的键：
- `COMPOSE_PROJECT_NAME=smartestate`
- `DB_NAME=smartestate_db`、`DB_USER=smartestate_user`、`DB_PASSWORD=smartestate_pwd`、`DB_ROOT_PASSWORD=smartestate_root`
- `JWT_SECRET=change_me_to_a_long_random_string`
- `FRONTEND_PORT=18412`、`BACKEND_PORT=19412`、`DB_PORT=3306`

Nginx 反向代理要求（`frontend/nginx.conf`）：
- `location /api/` 反向代理到 `http://backend:8080/`
- 配置 `try_files $uri $uri/ /index.html;` 支持前端 SPA 路由
